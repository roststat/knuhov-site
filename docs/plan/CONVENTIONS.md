# CONVENTIONS — единые решения для всех сессий

Обязателен к прочтению перед выполнением любого шага из `docs/plan/steps/`. Цель: независимые сессии с чистым контекстом принимают одинаковые решения. При конфликте: указания владельца > этот файл > docs/design-guide.md > здравый смысл.

## 1. Глоссарий и режим работы

- **«пуш»** (команда владельца) = полный релиз: `git add -A && git commit` → `git push origin main` → `vercel --prod --yes` (из корня репо `~/Downloads/knuhov-site`). Просто «git push» без деплоя командой «пуш» не считается.
- **По умолчанию НЕ деплоим и НЕ пушим** в конце шага. Коммитить локально — можно и нужно (один шаг = один коммит, сообщение на русском: `Sxx: краткая суть`). Релиз копится пачкой и уходит только по команде владельца «пуш».
- Один шаг за сессию. Сделал шаг → обновил `docs/plan/STATE.md` → остановился.
- Корневые `*.html` — артефакты сборки. Их не редактируют руками: правки в `src/pages/`, `src/partials/`, `build.js`, затем `node build.js`. Собранные файлы коммитятся вместе с исходниками (так задумано — Vercel раздаёт корень как есть).

## 2. Архитектура и данные

- Каталог и страницы услуг — data-driven: массив `SERVICES` и `CATEGORIES` в `build.js`. Контент услуги (заголовок, teaser, pain, includes, forWhom, цена, картинка) меняется ТОЛЬКО в данных, разметка — только в функциях-шаблонах (`cardHTML`, `servicePageMain`).
- Новые сквозные элементы head (canonical, OG, JSON-LD) добавляются в шаблон `HEAD`/`assemble()` в `build.js`, а не в отдельные страницы.
- Новые страницы = файл в `src/pages/*.html` с маркерами `<!--TITLE-->`, `<!--DESCRIPTION-->`, `<!--MAIN-->`, опционально `<!--SCRIPTS-->`.
- Не выдумывать факты бизнеса: цены, цифры, кейсы, реквизиты, телефоны. Если данных нет — шаг помечает «нужны данные от владельца» и использует нейтральную честную формулировку, а не фейк.

## 3. URL и ссылки

- После шага S01: внутренние ссылки — чистые, без `.html`: `/`, `/uslugi`, `/uslugi#tehnicheskie`, `/usluga-crm`, `/diagnostika`, `/cases`, `/contacts`. Главная — `/`, не `/index.html`.
- Канонический хост: `https://knuhov.ru` (без www). Абсолютные URL (canonical, OG, sitemap, JSON-LD) строить от константы `SITE_URL = 'https://knuhov.ru'` в `build.js`.
- Слаги — латиница-транслит в kebab-case, как существующие (`rabota-s-bazoy`).

## 4. Изображения

- **РЕШЕНИЕ ВЛАДЕЛЬЦА (2026-07-05, отменяет прежний ч/б-дефолт):** иллюстрации и графика — ЦВЕТНЫЕ, они намеренно разбавляют ч/б-каркас (кнопки, рамки, типографика остаются строго ч/б по design-guide). Палитра иллюстраций — на базе уже существующих в проекте цветов: красный `#d02026` (бренд), синий `#2563eb`, янтарный `#e0a526`, зелёный `#1f9d55` + их светлые тинты для фонов. Стиль: плоская геометрическая графика в духе фирменной мозаики — у крупных форм скруглён только нижний-правый угол. Не применять grayscale-фильтры к новым изображениям.
- **Хранение:** `assets/` в корне. Именование: `assets/usluga-<slug>.webp`, `assets/hero-<page>.webp`. Без вложенных папок, пока файлов < 30.
- **Форматы:** WebP основной; JPEG-fallback через `<picture>` только для хиро-изображений (LCP), для карточек достаточно `<img src="*.webp">` (поддержка WebP ~100%). Целевой вес: карточка ≤ 60 КБ, хиро ≤ 150 КБ.
- **Атрибуты обязательны:** `width`+`height` (реальные пропорции), `alt` осмысленный (для декоративных — `alt=""` + `aria-hidden="true"` на обёртке), `loading="lazy"` для всего ниже первого экрана, `fetchpriority="high"` (+`<link rel="preload">`) для LCP-хиро.
- **Пропорции:** визуал карточки — 4/3 (совпадает с `card-visual{aspect-ratio:4/3}`), хиро — свободно, но с фиксированными width/height.
- Инструменты конвертации: `cwebp -q 80` или `sips`/ImageMagick — что есть на машине; фиксировать в шаге фактическую команду.

## 5. Анимация

- Тонкая и служебная, «не эффектная простыня». Паттерны:
  - scroll-reveal: `opacity 0→1` + `translateY(16px)→0`, длительность 350–450 мс, easing `cubic-bezier(.22,.61,.36,1)`, срабатывает один раз, каскад внутри секции ≤ 3 элементов с шагом 80 мс.
  - hover: только инверсия ч/б и transform ≤ 4px, transition 150–200 мс (как существующие `.btn`, `.product-card`).
- **prefers-reduced-motion обязателен** для каждой новой анимации: CSS — внутри `@media (prefers-reduced-motion:no-preference)`; JS — проверка `matchMedia('(prefers-reduced-motion: reduce)').matches` → показать конечное состояние без анимации.
- Никаких анимационных библиотек — vanilla JS + CSS.
- JS-паттерн — IIFE в `src/partials/scripts-base.html` (сквозное) или в `<!--SCRIPTS-->` страницы (локальное), в стиле существующего кода (ES5-совместимо, без модулей).

## 6. SEO-разметка

- **title:** `{Суть страницы с ключом} — Кнухов консалтинг`, ≤ 60 символов до тире не обязательно, но без спама. Для услуг: `{Название услуги} для клиники — Кнухов консалтинг` (или естественная форма с «для клиник/медцентров»).
- **description:** 140–160 символов, содержит ключ + выгоду + призыв; уникален для каждой страницы.
- **h1:** ровно один на страницу, соответствует title по смыслу (не обязательно дословно). Дальше иерархия h2→h3 без пропусков.
- **canonical:** абсолютный, чистый URL без query/hash.
- **OG:** og:title (можно = title), og:description, og:type=website, og:url=canonical, og:image (1200×630, ч/б, дефолт `assets/og-default.png` для всех страниц, пока нет индивидуальных), og:locale=ru_RU, twitter:card=summary_large_image.
- **JSON-LD:** один `<script type="application/ld+json">` на сущность, вставляется шаблоном build.js. Идентификаторы: Organization `@id: https://knuhov.ru/#org`, Person `@id: https://knuhov.ru/#islam`, WebSite `@id: https://knuhov.ru/#website`. Не размечать то, чего нет на странице (рейтинги, отзывы, адрес до получения реквизитов).
- **sitemap.xml:** генерируется в `build.js` из списка собранных страниц; `case-*`-заглушки и служебные страницы (404, политика — политику включать) — см. конкретный шаг. `lastmod` — дата сборки.

## 7. Дизайн (сжатая выжимка design-guide + узаконенные исключения)

- Ч/б: #000/#fff, muted #555. Без теней и градиентов. Один шрифт Space Grotesk 400/500.
- Фирменный угол: только нижний-правый (`--radius` 24px / `--radius-sm` 12px / `--radius-xs` 8px). Крупные инверсные блоки (манифест, футер) зеркалят угол — смотри существующие.
- Новые блоки собирать из готовых классов (`.container/.container--wide`, `.btn`, `.kicker`, `.lead`, `.product-card`, `.audit-steps`, `.benefits`) — новый CSS только если примитива нет.
- **Выравнивание по 3-полосной сетке:** новые секции выравниваются по той же 3-колоночной сетке, что карточки каталога (`repeat(3,1fr)`, gap 16px), а не на всю ширину (правило владельца; образцы: `.trust-grid`, `.bridge-grid`).
- Узаконенные цветовые исключения (не расширять без владельца): красный `#d02026` только в «Dr.» логотипа; синий `#2563eb` тултипа «слои»; светофор `#d02026/#e0a526/#1f9d55` только внутри отчёта диагностики; цветное фото в хиро /diagnostika.
- Голос: «мы» — про исполнение, «я» (Ислам) — про метод и мышление. Слово «агентство» запрещено; sentence case.

### 7.1. Тёмная тема (добавлена 2026-07-06, вне исходного плана из 36 шагов)

Сайт поддерживает переключаемую тёмную тему — кнопка-иконка (солнце/луна) в шапке, `.theme-toggle` в `src/partials/header.html`, обёрнута вместе с `.nav-cta` и `.nav-burger` в `.header-actions`.

- **Механизм:** атрибут `data-theme="dark"` на `<html>`. Инлайн-скрипт в начале `<head>` (в шаблоне `HEAD` в `build.js`) выставляет атрибут ДО отрисовки — на основе `localStorage['knuhov_theme']`, а если ключа нет — на основе `prefers-color-scheme`. Клик по `.theme-toggle` (обработчик в `scripts-base.html`) переключает атрибут и сохраняет выбор в `localStorage`.
- **Токены-переключатели** (объявлены в `:root`, переопределены в `html[data-theme="dark"]`): `--ink`, `--paper`, `--muted`, `--line`, `--hair`, `--hover-tint`. Использовать ИХ для любого нового компонента, где фон/текст должны корректно инвертироваться между темами (это подавляющее большинство случаев — кнопки, карточки, поля, попапы).
- **Фиксированные токены** (НЕ зависят от темы, буквально `#000`/`#fff` всегда): `--true-ink`, `--true-paper`. Использовать ТОЛЬКО там, где элемент обязан оставаться буквально чёрным/белым независимо от темы страницы — сейчас это подвал (`.footer`, `.footer .help`, `.social a`, `.big-link`, `--light`-модификаторы лого) и `.manifesto--center` (цитата на сплошном чёрном). **Не путать эти две группы токенов** — если новый компонент случайно получит `var(--ink)` там, где нужен `var(--true-ink)` (или наоборот), при переключении темы либо подвал станет белым, либо новый компонент не будет реагировать на тему.
- Брендовые цветовые исключения (`#d02026`, тултип `#2563eb`, светофор диагностики) — фиксированные литералы, тему не учитывают, трогать не нужно.
- Все новые transition, связанные с сменой темы — внутри `@media (prefers-reduced-motion:no-preference)`, как и остальные анимации (см. §5).
- Проверка нового компонента на тёмную тему: `document.documentElement.setAttribute('data-theme','dark')` в консоли (или клик по `.theme-toggle`) и визуально свериться, что текст читаем и подвал остался чёрным.

## 8. Аналитика

- Все новые интерактивы обязаны звать `window.track('event_name',{payload})` (снейк-кейс, по образцу существующих `diagnostic_*`). Метрика подключается в `track()` централизованно (шаг S32) — события в коде страниц не меняются.

## 9. Проверка шага (общая для всех)

1. `node build.js` — без ошибок, в консоли `built ...` по каждой странице.
2. Открыть затронутые корневые `.html` локально (`npx serve .` или `python3 -m http.server` из корня; просто file:// не годится из-за абсолютных путей `/styles.css`).
3. Проверить Definition of Done шага, мобильную ширину (≤820px) и prefers-reduced-motion, если менялись вёрстка/анимации.
4. `git diff` — в диффе нет случайных файлов; собранные .html изменились ожидаемо.
