# S36 — Яндекс.Вебмастер + Google Search Console

## 1. Цель
Подтвердить права на knuhov.ru в Яндекс.Вебмастере (приоритет) и Google Search Console, отправить sitemap, убедиться в отсутствии блокеров индексации. Это операционный шаг «после деплоя фундамента», а не правка кода (кроме файла/метатега верификации).

## 2. Контекст-бутстрап
Проект: knuhov.ru — статический сайт «Кнухов консалтинг» (SEO-приоритет — Яндекс). Плоский HTML/CSS/JS, сборка `node build.js`; сквозной `<head>` — шаблон `HEAD` в `build.js` (мета-верификация добавляется туда); файлы в корне репо раздаются как есть (HTML-файл верификации можно просто положить в корень). robots.txt и sitemap.xml существуют (S07). Домен проксируется на Vercel через VPS — на выполнение шага не влияет, но если Вебмастер покажет проблемы доступности, причина может быть в прокси.
Прочитай первыми: `docs/plan/README.md`, `docs/plan/CONVENTIONS.md`, `build.js` (шаблон HEAD), `robots.txt`, `sitemap.xml`, `docs/plan/STATE.md` (что уже задеплоено).
Глоссарий: «пуш» (команда владельца) = git add+commit → `git push origin main` → `vercel --prod --yes`. Без команды «пуш» не деплоить.

## 3. Предусловия/зависимости
S07 — done И **задеплоен в прод** (sitemap должен быть доступен по https://knuhov.ru/sitemap.xml — проверь curl'ом). **Нужен доступ владельца**: аккаунт Яндекса и Google (владелец добавляет сайт сам под твою диктовку, либо даёт делегированный доступ). Желательно, чтобы Волна 1 целиком была в проде — иначе Вебмастер сразу насобирает уже исправленных ошибок.

## 4. Затрагиваемые файлы
- `build.js` (мета-теги верификации в HEAD) ИЛИ файлы верификации в корне (`yandex_*.html`, `google*.html`)
- корневые `*.html` — пересборка (если мета-способ)

## 5. Точные инструкции
1. **Яндекс.Вебмастер** (webmaster.yandex.ru): добавить сайт `https://knuhov.ru`; верификация — мета-тег в шаблон HEAD (`<meta name="yandex-verification" content="…">`) или HTML-файл в корень. После подтверждения: Индексирование → Файлы Sitemap → добавить `https://knuhov.ru/sitemap.xml`; проверить «Диагностика сайта» (фатальные/критичные); Индексирование → Переобход — отправить главную и /uslugi; регион НЕ сужать (бизнес всероссийский, онлайн).
2. **Google Search Console**: property `https://knuhov.ru` (URL-prefix проще для верификации метатегом `google-site-verification`); Sitemaps → отправить; URL Inspection главной — Request indexing.
3. Если выбран мета-способ: добавить оба метатега в `HEAD` build.js, `node build.js`, закоммитить — и для проверки поисковиками потребуется деплой (см. §8: это тот случай, когда нужно попросить у владельца команду «пуш» именно для верификации, не деплоить самовольно).
4. Зафиксируй в STATE: даты подтверждения, статус sitemap (принят/ошибки), найденные Вебмастером проблемы — как кандидатов в новые шаги.

## 6. Definition of Done
- [ ] Оба сервиса: права подтверждены, sitemap принят без ошибок.
- [ ] «Диагностика сайта» Вебмастера без фатальных/критичных (или они выписаны в STATE как кандидаты в шаги).
- [ ] robots.txt распознан обоими сервисами.

## 7. Как проверить локально
Локальной проверки почти нет (шаг операционный): `curl -I https://knuhov.ru/sitemap.xml` → 200; `curl https://knuhov.ru/robots.txt` → содержит Sitemap-строку; если добавлял метатеги — `grep verification index.html` после сборки.

## 8. Деплой
Метатеги верификации — единственное исключение, когда деплой нужен внутри шага: запроси у владельца команду «пуш» явно. Сам по умолчанию НЕ деплой: `git push origin main && vercel --prod --yes` — только по его команде.

## 9. Финализация
Коммит `S36: верификация Вебмастер/GSC` (если были правки кода). Обновить `docs/plan/STATE.md` (S36 → done + даты/статусы, Журнал). Остановиться.
