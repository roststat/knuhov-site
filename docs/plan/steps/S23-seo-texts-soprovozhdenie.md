# S23 — SEO-тексты услуг категории «Сопровождение» (6 услуг)

## 1. Цель
Наполнить страницы услуг категории «Сопровождение» до 300–500 слов каждая: процесс, FAQ, расширенный текст — через поля данных в `SERVICES` (шаблон готов в S22). Цель — и убеждение, и покрытие запросов «{услуга} для клиники».

## 2. Контекст-бутстрап
Проект: knuhov.ru — статический сайт «Кнухов консалтинг» (консалтинг для клиник; голос: «мы» — исполнение, «я» (Ислам) — мышление; слово «агентство» запрещено; тексты money-ориентированные — про выручку и потери, без воды). Плоский HTML/CSS/JS. Контент услуг живёт в массиве `SERVICES` в `build.js` (поля pain/includes/forWhom + новые process/faq/seoText из S22); правки — только в данных, `node build.js` пересобирает страницы. Услуги категории `soprovozhdenie`: rabota-s-bazoy, proektnyy-menedzher, reklamshchik, hr, smm, taynyy-pokupatel.
Прочитай первыми: `docs/plan/README.md`, `docs/plan/CONVENTIONS.md` (§2, §6), `docs/audit/SEO.md` (§J — кластеры), `docs/TZ.md` (§1 — голос), `build.js` (SERVICES + шаблон), собранный `usluga-crm.html` (образец заполненного шаблона из S22).
Глоссарий: «пуш» (команда владельца) = git add+commit → `git push origin main` → `vercel --prod --yes`. Без команды «пуш» не деплоить.

## 3. Предусловия/зависимости
S22 — done (шаблон поддерживает process/faq/seoText).

## 4. Затрагиваемые файлы
- `build.js` (только данные `SERVICES` — 6 объектов категории soprovozhdenie)
- корневые `usluga-rabota-s-bazoy.html`, `usluga-proektnyy-menedzher.html`, `usluga-reklamshchik.html`, `usluga-hr.html`, `usluga-smm.html`, `usluga-taynyy-pokupatel.html` — пересборка

## 5. Точные инструкции
1. Для каждой из 6 услуг заполни: `process` (3–4 шага: как начинается работа → что делаем → как отчитываемся; логика из `includes`), `faq` (3–5 вопросов: сроки, формат работы, что нужно от клиники, как меряется результат, чем отличается от найма в штат), `seoText` (2–3 абзаца, 120–200 слов: разворот pain → как устроена услуга → связка с системой/диагностикой).
2. Правила текста: естественные вхождения ключей («работа с базой пациентов», «HR для клиники», «SMM для стоматологии/клиники» и т.п. — по карте SEO.md §J), без переспама; никаких выдуманных цифр, кейсов, сроков-обещаний и гарантий («увеличим на 30%» — нельзя; «покажем в отчёте, сколько выручки дала база» — можно); голос «мы»; sentence case; терминология категорий сайта.
3. Черновой статус: тексты услуг помечены в build.js как «требует согласования с Исламом» (комментарий вверху файла) — сохрани этот статус-кво: новые тексты тоже черновик до вычитки владельцем; отметь в STATE.
4. `node build.js`; вычитай 2 собранные страницы глазами (опечатки, вёрстка details).

## 6. Definition of Done
- [ ] Все 6 услуг категории имеют process + faq + seoText; страницы 300–500 слов.
- [ ] FAQ рендерится и попадает в FAQPage-разметку (если механика S22/S10 включена).
- [ ] Ни одного выдуманного факта/цифры/гарантии; голос и лексика соблюдены.
- [ ] Правки только в данных SERVICES (шаблон не менялся).

## 7. Как проверить локально
`node build.js`; `npx serve .` → пройти все 6 страниц (десктоп + 375px); выборочно подсчитать объём: видимый текст ≥ 300 слов.

## 8. Деплой
По умолчанию НЕ деплоить: изменения уходят пачкой по команде владельца «пуш». Если скомандовал: `git push origin main && vercel --prod --yes`.

Коммит `S23: тексты услуг «Сопровождение»`. Обновить `docs/plan/STATE.md` (S23 → done + заметка «тексты — черновик, нужна вычитка Ислама», Журнал). Остановиться.
