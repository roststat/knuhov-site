# S19 — Политика конфиденциальности + рабочие ссылки футера (152-ФЗ)

## 1. Цель
Сайт собирает ПДн через форму (имя, телефон/Telegram), а политики конфиденциальности нет — ссылка в футере ведёт на `#`. Создать страницу политики, починить ссылки футера, связать чекбокс согласия с политикой.

## 2. Контекст-бутстрап
Проект: knuhov.ru — статический сайт «Кнухов консалтинг» (консалтинг для клиник Ислама Кнухова). Плоский HTML/CSS/JS, сборка `node build.js` из `src/pages/*.html` + партиалы. Форма с чекбоксом согласия — `src/pages/contacts.html` (`label.consent`); ссылки-заглушки — `src/partials/footer.html`, блок `.footer-bottom` («Политика конфиденциальности» и «Условия» → `href="#"`). Корневые `*.html` — артефакты сборки.
Прочитай первыми: `docs/plan/README.md`, `docs/plan/CONVENTIONS.md`, `src/partials/footer.html`, `src/pages/contacts.html`, `src/pages/cases.html` (образец простой страницы), `docs/plan/STATE.md` (получены ли реквизиты в S18).
Глоссарий: «пуш» (команда владельца) = git add+commit → `git push origin main` → `vercel --prod --yes`. Без команды «пуш» не деплоить.

## 3. Предусловия/зависимости
S18 (реквизиты оператора ПДн: наименование ИП, ИНН, ОГРНИП, email). Если реквизитов ещё нет — страницу можно свёрстать целиком с помеченными плейсхолдерами В КОММЕНТАРИЯХ сборки не видимых пользователю нельзя; тогда шаг остаётся blocked до данных, публиковать с «[Черновик]» запрещено.

## 4. Затрагиваемые файлы
- `src/pages/politika-konfidencialnosti.html` (создать)
- `src/partials/footer.html` (ссылки нижней строки)
- `src/pages/contacts.html` (текст согласия со ссылкой)
- `build.js` — только если нужно исключить страницу из чего-либо (в sitemap её ВКЛЮЧАТЬ)
- корневые `*.html` — пересборка

## 5. Точные инструкции
1. Создай `src/pages/politika-konfidencialnosti.html` (TITLE: `Политика конфиденциальности — Кнухов консалтинг`; DESCRIPTION нейтральный). Структура MAIN: h1 «Политика обработки персональных данных», далее секции текстом (класс `.container`, обычная типографика, без карточек): 1) Оператор (реквизиты из S18); 2) Какие данные собираем (имя, контакт — телефон/Telegram, содержимое сообщения; технические данные метрики — добавить после S32); 3) Цели (связь по заявке, договор); 4) Правовое основание (согласие, 152-ФЗ); 5) Хранение и защита (где хранятся заявки — сверить с S34, до его выполнения — «на защищённой инфраструктуре хостинг-провайдера»); 6) Передача третьим лицам (не передаём, кроме требований закона); 7) Права субъекта и контакт для запросов (email); 8) Дата редакции. Текст юридически стандартный, без выдуманных фактов; формулировки, зависящие от невыясненных решений, — максимально нейтральные.
2. Футер: «Политика конфиденциальности» → `/politika-konfidencialnosti`. «Условия» — отдельной страницы нет и не планируется: удали эту ссылку (вместе с её `.sqdot`), не оставляй пустышку.
3. Согласие в форме контактов: `Согласен на обработку персональных данных` → `Согласен на <a href="/politika-konfidencialnosti" target="_blank">обработку персональных данных</a>`.
4. `node build.js`. Страница попадёт в sitemap автоматически (S07) — это правильно.

## 6. Definition of Done
- [ ] Страница `/politika-konfidencialnosti` собрана: один h1, реальные реквизиты оператора, 8 секций.
- [ ] В футере нет `href="#"`; «Условия» удалены.
- [ ] Чекбокс согласия ссылается на политику.

## 7. Как проверить локально
`node build.js`; `npx serve .` → открыть страницу политики (десктоп/375px), клик из футера и из чекбокса формы; `grep -rn 'href="#"' src/partials/footer.html` — пусто.

## 8. Деплой
По умолчанию НЕ деплоить: изменения уходят пачкой по команде владельца «пуш». Если скомандовал: `git push origin main && vercel --prod --yes`.
