## A. Информационная архитектура, навигация, глубина

**Находки:**
1. Структура здоровая и уже НЕ лендинг: главная → каталог `/uslugi.html` (фильтр по 3 категориям, deep-link через `#diagnostika|#soprovozhdenie|#tehnicheskie`) → 11 отдельных страниц услуг → `/diagnostika.html` (лид-магнит) → кейсы → контакты. Каталог data-driven (массив `SERVICES` в `build.js:31-96`) — добавление услуги = добавление объекта. Это сильная база, её надо беречь и углублять.
2. **Мобильной навигации нет вообще**: `styles.css:387` — `.nav,.nav-cta{display:none}` при ≤820px, бургер не предусмотрен. С телефона (а ТЗ прямо говорит «мобайл — приоритет, тёплый трафик с телефона») пользователь может уйти со страницы только через логотип или футер. **P0.**
3. Страницы услуг — тупиковые ветки: единственный выход — кнопка «Оставить заявку» на контакты. Нет перелинковки «смежные услуги», нет хлебных крошек, нет возврата в каталог. Ослабляет и UX, и SEO. **P1.**
4. В шапке нет пункта «Экспресс-диагностика» в основной навигации (есть только CTA-кнопка, скрытая на мобильных) — на мобильных лид-магнит недостижим из меню. **P1** (решается вместе с бургером).
5. Нет страницы «Об Исламе» — при этом весь бренд эпонимный, ТЗ строит продажу через личность. Сейчас личность живёт только блоком «Опыт» на главной и диагностике. **P1.**
6. Риск «лендинга» на данный момент низкий; риск обратный — тонкие страницы услуг (см. C).

**Влияние:** мобильные пользователи (приоритетная аудитория) фактически заперты на первой странице; глубина сайта не работает на удержание.

2. Осознанные отступления от ч/б, НЕ зафиксированные в гайде: синий `#2563eb` у тултипа «слои» (`styles.css:336`), цвета отчёта диагностики `#d02026/#e0a526/#1f9d55` (`src/pages/diagnostika.html:428-432` в `colorForScore`) и жёлтый значок `#e0a526` (`diagnostika.html:226`). Надо либо узаконить в гайде/конвенциях, либо перевести в ч/б. **P2** (зафиксировано в CONVENTIONS как узаконенное исключение внутри попапа).

5. Инлайн-стили (`style="text-align:center"`, `style="margin:14px auto 0"` и т.п.) рассыпаны по страницам — терпимо для текущего масштаба, но новые правки должны идти в классы. **P2.**

## D. Медиа: изображения

**Находки:**
1. Всего 2 изображения на весь сайт: `assets/hero-bg.jpg` (552 КБ — не оптимизировано, только на /diagnostika) и `assets/islam.jpg` (84 КБ). Главная — вообще без изображений в хиро.
2. У `<img>` нет `width/height` → сдвиги макета (CLS); нет `loading="lazy"` у подвальных/нижних изображений; нет WebP/AVIF; нет `preload`/`fetchpriority` для LCP-картинки на /diagnostika.
3. Карточки услуг: `card-visual` — заготовка под визуал, сейчас пустая. Владелец хочет картинки на карточках. Через data-driven архитектуру это делается правильно: поле `image` в объекте `SERVICES` → `cardHTML()` (`build.js:107-117`) рендерит `<img>`.
4. Хиро страниц услуг — чистый текст; владелец хочет визуалы в хиро ключевых страниц.
5. Обработка ч/б уже отработана: `filter:grayscale(1) contrast(1.05)` на hero-bg (`styles.css:99`) — этот приём масштабируется на все будущие изображения.

## E. Анимация и интерактив

**Находки:**
1. Уже есть: анимация-печаталка логотипа (с корректным `prefers-reduced-motion` фолбэком, `styles.css:75-77`), счётчик цифр в «Опыте» (`src/partials/scripts-base.html:10-36`), blink кнопки, попап диагностики, фильтр каталога. Интерактивная диагностика — сильный актив.
2. **Счётчик цифр игнорирует `prefers-reduced-motion`** — JS-анимация через rAF без проверки медиа-запроса. CSS-правило `*{transition:none!important}` (`styles.css:398`) на rAF не действует. **P1.**
3. Секции появляются без какого-либо scroll-reveal — владелец хочет аккуратной анимации. Уместный паттерн: подъём+fade секций через IntersectionObserver, 300–400 мс, один раз, с полным отключением при reduced-motion. **P1.**
4. Тултип «слои» работает по hover/focus — на тач-устройствах hover нестабилен, но `tabindex="0"` даёт фолбэк. Приемлемо. **P2.**
5. Попап: Escape и клик по оверлею работают, но нет focus-trap и возврата фокуса на открывшую кнопку. **P2** (a11y, см. G).

## F. Производительность (Core Web Vitals)

**Находки:**
1. `hero-bg.jpg` 552 КБ — LCP-элемент /diagnostika. Пережать/WebP → ожидаемо ≤120 КБ. **P1.**
2. Шрифты с fonts.googleapis.com: два preconnect есть, `display=swap` есть, но (а) внешний CSS-запрос блокирует отрисовку, (б) Google Fonts в РФ периодически деградирует. Самохостинг woff2 (2 файла: 400, 500) + `preload` — быстрее и надёжнее для РФ-аудитории. **P1.**
3. Нет `width/height` на изображениях → CLS. **P1.**
4. Анимация-печаталка `brand-kicker` анимирует `width` → постоянные reflow в шапке; на слабых устройствах может влиять на INP. Заменить на `clip-path`/`transform`-подход или оставить осознанно. **P2.**
5. Всё остальное лёгкое: CSS 440 строк одним файлом, JS инлайновый и маленький, без сторонних библиотек. База для «зелёных» CWV отличная.

## G. Доступность (a11y)

**Находки:**
1. Семантика в целом хорошая: `<header>/<main>/<footer>/<nav aria-label>`, aria-hidden на декоре, `role="dialog" aria-modal` на попапе, labels у полей формы.
2. **Нет h1 на большинстве страниц** (см. SEO.md §H.3) — это и a11y-проблема (навигация скринридером по заголовкам).
3. Мобильная навигация отсутствует (см. A.2) — для a11y критично так же, как для UX.
4. Форма контактов: поля без `required`/валидации, отправка не блокируется без согласия ПДн, «спасибо»-блок без `aria-live`, `<input>` для сообщения вместо `<textarea>`. **P1.**
5. Попап без focus-trap; фокус не возвращается на триггер при закрытии. **P2.**
6. Контраст: чёрный на белом — максимальный; `--muted:#555` на белом ≈ 7.4:1 — ок. Серый `rgba(255,255,255,.5)` на чёрном в футере ≈ 10:1 по крупному тексту — ок. Жёлтый `#e0a526` на белом в отчёте диагностики ≈ 2.1:1 — ниже нормы для значимого текста. **P2.**
7. Ссылки-заглушки `href="#"` дезориентируют пользователей скринридеров. **P0** вместе с C.7.

**Находки:**

При этом слой событий уже построен: `window.track()` (`src/partials/scripts-base.html:3-9`) пушит в `dataLayer`, и события уже расставлены по всем ключевым действиям: `diagnostic_start`, `diagnostic_step_answered`, `diagnostic_completed`, `audit_cta_clicked`, `diagnostic_share_*`, `diagnostic_pdf_*`, `diagnostic_attached_to_audit`, `contact_form_submitted`.

3. Данных владельца: номер счётчика Метрики (или создать). GA4 — опционально вторым слоем.
